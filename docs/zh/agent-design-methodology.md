# 面向 Agent 项目的可迁移设计方法论

> 这份文档面向后续 AI 复用。它不复述项目实现，也不作为项目说明书使用，而是把本项目中可迁移的 Agent 架构方法提炼出来：为什么要这样拆分模块，模块之间如何协作，哪些设计原则适合迁移到新的 Agent 项目中。

## 设计命题：把模型放进可治理的工作环境

一个可用的 Agent 系统，核心问题不是“让模型多聪明”，而是为模型提供一个可持续工作的环境。

模型负责推理、规划和选择动作；工程系统负责提供状态、能力、安全边界、执行环境、恢复机制和经验沉淀。这个工程系统可以理解为 Agent 的 harness。它把模型的一次次输出转换成可执行、可恢复、可审计、可复用的工作流。

因此，本项目体现出的主要设计命题是：

- Agent 不应只是一个对话循环，而应是“入口、循环、能力、执行、持久化、演进”共同构成的平台。
- 模型上下文不是唯一状态；工作记忆、会话记录、长期记忆、配置、技能和评估数据都应分层管理。
- 能力扩展不应侵入核心循环；工具、技能、外部协议、插件和 hook 应有不同边界。
- 安全与恢复不是外围功能，而是工具执行和模型调用路径上的一等分支。
- Agent 的长期价值来自运行后的沉淀：把一次任务中的有效经验变成记忆、技能、测试样例或优化反馈。

## 总体方法：稳定内核，边界适配，经验闭环

可迁移的总体结构可以抽象为三层策略。

```text
入口与边界层
  统一来自 CLI、消息平台、定时触发、外部服务的输入
  隔离平台协议、消息格式、执行位置和外部依赖

Agent 内核层
  保持一个稳定的对话循环
  负责模型调用、工具分发、上下文管理、错误恢复和安全关卡

经验与演进层
  将会话沉淀为记忆、技能、评估数据和优化反馈
  通过后台审视、hook、插件和评估优化机制持续改进系统
```

这套结构的关键不是层数本身，而是职责方向：入口层向内规范化，能力层向外抽象化，持久层按生命周期保存状态，演进层把运行经验重新喂回系统设计。

## 一、核心循环：让所有入口共享同一个工作心脏

### 解决的问题

Agent 很容易从一个单入口对话程序膨胀成多个重复系统：CLI 一套逻辑，平台消息一套逻辑，定时任务一套逻辑，外部触发又一套逻辑。这样会导致工具调用、记忆、权限和错误处理分散在不同入口里，后续维护成本迅速上升。

### 设计方法

保留一个唯一的核心循环：接收用户意图，组装上下文，调用模型，执行工具，把工具结果写回上下文，继续直到任务完成或预算耗尽。

不同入口不直接实现 Agent 逻辑，而是先转换成统一的消息事件，再交给核心循环。回复也沿相反方向由入口适配器转换回平台格式。

### 模块协作

- 入口适配器负责接收外部消息、转换格式、发送回复。
- 会话路由负责根据来源识别对应会话，并加载必要历史。
- 核心循环只面对统一消息和统一工具接口。
- 持久化层负责记录消息、工具结果、会话元信息和后续检索需要的内容。

### 可迁移经验

新 Agent 项目应优先确认“核心循环是否只有一个”。如果不同入口各自复制循环逻辑，后续每增加一个安全策略、工具协议或上下文策略都要重复改多处。入口可以很多，心脏应只有一个。

## 二、状态分层：按生命周期和写入责任拆分上下文

### 解决的问题

很多 Agent 项目把所有信息都塞进一个 messages 列表：当前任务、历史会话、用户偏好、项目规则、工具输出、长期记忆混在一起。短期看简单，长期会造成上下文膨胀、状态污染、重启丢失和不可审计。

### 设计方法

将状态按生命周期拆分，而不是按技术载体拆分。

- 工作上下文：当前轮对话需要模型立即看到的信息。
- 会话记录：完整保存一次任务过程，用于恢复、检索和审计。
- 长期记忆：跨会话仍有价值的精选事实、偏好和经验。
- 项目规则：来自当前工作区或任务域的约束。
- 技能资产：可复用的操作方法、流程和注意事项。
- 配置档案：运行时策略、能力开关、模型与环境选择。

### 模块协作

Prompt 组装器负责把长期记忆、项目规则、技能索引和临时上下文整理成模型可读输入；会话存储保存完整轨迹；记忆管理只保存未来仍有用的精选内容；技能系统保存可复用方法，而不是保存一次对话的流水账。

### 设计原则

状态拆分应遵循三个问题：

- 这条信息下轮是否必须看到？
- 这条信息重启后是否必须保留？
- 这条信息应该由系统写入、用户写入，还是 Agent 主动沉淀？

能回答这三个问题，就能避免把所有状态都塞进 prompt 或都塞进数据库。

## 三、提示词组装：从硬编码大段文本转向分区合成

### 解决的问题

系统提示词会逐渐承载人设、规则、记忆、工具说明、技能索引和项目约束。如果用一个大字符串维护，任何修改都会影响整体缓存、难以定位来源，也容易把过多无关内容塞给模型。

### 设计方法

把提示词看成多个来源的合成结果，而不是单一文件。每个来源有明确职责、优先级、截断策略和缓存策略。

关键思想是：提示词不是“越全越好”，而是“让模型在当前任务中看到必要、稳定、边界清晰的信息”。

### 模块协作

- 配置和项目规则提供行为边界。
- 记忆层提供跨会话精选信息。
- 技能层只暴露技能索引，具体内容按需读取。
- 工具注册表提供当前可用工具的能力描述。
- 压缩层在上下文过长时保留任务状态，而不是盲目删除历史。

### 可迁移经验

提示词系统应从一开始就支持分区。即使早期只有两三个来源，也应保留来源边界。后续加入记忆、技能、项目规则和插件上下文时，系统不会退化成不可维护的“巨型 prompt”。

## 四、工具体系：把能力注册和能力执行解耦

### 解决的问题

工具数量少时，可以用条件分支直接分发。工具数量增长后，分发逻辑会变成中心化瓶颈：添加工具要改核心循环，外部工具要改注册代码，异步能力和同步循环之间也会互相污染。

### 设计方法

建立工具注册表，工具自描述并自注册。核心循环只按工具名查表分发，不关心工具来自内置代码、外部协议还是插件。

工具体系至少应拆成三件事：

- 给模型看的能力描述。
- 系统内部执行的处理函数。
- 注册、筛选、分发和可用性检查。

### 模块协作

- 工具注册表维护工具目录。
- 编排层负责加载工具模块和外部工具来源。
- 核心循环只调用统一分发接口。
- 权限系统在高风险工具的执行路径上插入检查。
- 外部协议桥接层把外部能力映射为同一类工具。

### 可迁移经验

工具扩展应满足一个简单检验：新增一种能力时，核心循环不需要修改。若添加工具必须改循环，说明能力边界还没有抽出来。

## 五、技能体系：把“怎么做事”从硬编码能力中分离出来

### 解决的问题

工具回答“系统能做什么”，但不能回答“面对某类任务时应该怎样组织步骤、规避坑点、检查结果”。如果所有经验都写进系统提示词，prompt 会膨胀；如果都写进代码，Agent 无法自行改进。

### 设计方法

引入技能资产：它不是工具，也不是记忆，而是一种可读、可编辑、可复用的方法文档。工具是执行能力，技能是使用能力的方法。

技能系统的关键是渐进展示：模型先看到技能目录和简短说明，只有在需要时再读取具体技能内容。这样既保留可发现性，又避免把所有技能全文塞入上下文。

### 模块协作

- 技能发现模块扫描可用技能并生成索引。
- Prompt 组装器注入技能目录，而不是注入全部正文。
- 技能读取工具按需提供细节。
- 技能管理工具允许 Agent 在受控路径下创建、更新或修补技能。
- 后台审视模块从对话中发现可沉淀的模式，并推动技能更新。

### 可迁移经验

技能适合保存“可迁移流程”，不适合保存“某次任务的事实”。事实进入记忆，流程进入技能，执行能力进入工具。三者边界清楚，Agent 才能越用越稳。

## 六、安全设计：把高风险动作放在明确关卡上

### 解决的问题

模型会提出行动，但不能让所有行动自动变成系统副作用。尤其是终端、文件写入、远程执行、数据库操作和外部服务调用，一旦误判可能造成不可逆后果。

### 设计方法

安全系统不必一开始追求覆盖所有工具，而应先识别最高风险路径，在那里插入强约束。终端执行通常是最高风险点，因此适合作为第一道安全关卡。

有效的安全设计包括：

- 对高风险命令做模式识别。
- 提供一次、会话级、长期级的审批粒度。
- 对无响应或异常审批默认拒绝。
- 对执行环境做密钥隔离和资源限制。
- 对子任务和子 Agent 限制工具集和持久副作用。

### 模块协作

权限系统应靠近工具执行点，而不是散落在模型输出解析里。核心循环知道“工具被调用”，但具体某个工具是否需要审批，应由工具或工具前置层决定。

### 可迁移经验

不要把安全设计做成空泛的“所有工具都问一下”。这会让系统不可用。先保护最危险、最不可逆、最容易被模型误用的能力，再逐步扩展策略。

## 七、错误恢复：把失败分类为下一步策略

### 解决的问题

真实 Agent 会遇到模型输出截断、上下文超限、网络抖动、限流、认证失败、模型不可用和工具异常。如果所有错误都只是抛出异常，Agent 会在可恢复问题上提前失败；如果所有错误都盲目重试，又会造成死循环和成本浪费。

### 设计方法

先分类，再恢复。错误分类器把底层错误翻译成策略信号：是否重试、是否压缩、是否切换备用能力、是否续写、是否终止。

### 模块协作

- 模型调用层捕获并分类 API 错误。
- 上下文管理层处理超限与压缩。
- 模型路由层处理备用模型或备用服务。
- 核心循环维护迭代预算、重试预算和续写预算。
- 持久化层保留失败前状态，方便恢复和审计。

### 可迁移经验

错误恢复不是一个 try-catch，而是一张策略表。每类失败都应有明确的后续动作和预算上限。预算是防止恢复逻辑变成无限循环的关键。

## 八、上下文压缩：保留任务状态，而不是简单删历史

### 解决的问题

Agent 做长任务时，上下文一定会膨胀。粗暴删除旧消息会丢失决策、已尝试方案、文件状态和用户约束；完全不压缩则会触发上下文上限或拖慢模型。

### 设计方法

压缩应把历史转化为任务状态摘要。摘要重点保留目标、进度、关键决策、已修改对象、剩余工作和风险，而不是生成泛泛的聊天总结。

压缩还应保护最近上下文，因为最新工具结果和用户指令通常最容易被后续步骤直接依赖。

### 模块协作

- 核心循环在调用模型前检查上下文规模。
- 压缩层裁剪低价值的大型工具输出。
- 摘要模型生成中段历史摘要。
- 会话存储保留压缩前完整轨迹，并把压缩后会话与原始轨迹关联。
- Prompt 组装器在压缩后重建稳定上下文。

### 可迁移经验

压缩不是删除机制，而是状态迁移机制。一个好压缩结果应该让 Agent 能继续做事，而不是只让人知道之前聊过什么。

## 九、边界适配：入口适配器和执行后端必须分开

### 解决的问题

跨平台 Agent 同时面对两个完全不同的问题：消息从哪里来，命令在哪里跑。很多系统会把这两个问题混在一起，导致平台适配器知道执行环境，工具知道消息平台，核心循环被边界细节污染。

### 设计方法

把入口适配和执行环境分成两个独立抽象。

入口适配器负责消息协议：连接平台、接收消息、格式转换、去重、合并、媒体缓存、回复发送。

执行后端负责动作位置：本地、容器、远程机器或云端环境。工具只发出抽象执行请求，不关心命令在哪运行。

### 模块协作

- Gateway 聚合入口适配器输出的统一消息事件。
- 会话路由根据来源隔离会话。
- 工具层调用执行环境接口。
- 执行环境基类处理通用状态延续、超时、清理和密钥隔离。
- 具体后端只实现“如何执行”。

### 可迁移经验

新增消息平台不应影响工具执行；新增执行环境不应影响消息接入。这两个维度正交，是跨平台 Agent 保持可维护性的关键。

## 十、子 Agent 委派：用隔离上下文处理局部复杂度

### 解决的问题

复杂任务会产生大量中间搜索、文件读取、实验和验证结果。如果全部进入父对话，主上下文会被噪声污染。独立子任务之间也会互相干扰。

### 设计方法

子 Agent 用独立上下文完成局部任务，只把最终结论返回父 Agent。父 Agent 保持主线清晰，子 Agent 承担探索、调研、局部验证等临时工作。

### 模块协作

- 父 Agent 通过委派工具创建子 Agent。
- 子 Agent 继承必要的模型配置和任务上下文。
- 子 Agent 的工具集受限，避免递归委派、用户交互和持久记忆修改。
- 父子共享迭代预算，避免无限扩张。
- 子 Agent 完成后销毁，中间轨迹不回灌父上下文。

### 可迁移经验

委派不是为了“看起来多 Agent”，而是为了上下文隔离、并行探索和风险控制。适合委派的是边界清楚、结果可汇总、失败可容忍的子任务。

## 十一、后台审视：把运行经验转化为长期资产

### 解决的问题

用户不会总是说“请把这个经验记下来”或“请沉淀成技能”。如果 Agent 只能被动等待显式指令，很多有价值的偏好、坑点和流程会随着会话结束而消失。

### 设计方法

引入后台审视机制：在对话达到合适边界后，用隔离的审视任务检查是否有值得长期保存的内容。审视不是聊天总结，而是经验筛选：哪些应进入记忆，哪些应变成技能，哪些不应保存。

### 模块协作

- 主 Agent 完成任务后提供会话快照。
- 后台审视 Agent 在隔离环境中分析快照。
- 记忆系统接收可复用事实和偏好。
- 技能系统接收可重复流程和注意事项。
- 审视失败不影响主任务结果。

### 可迁移经验

后台审视应是 best-effort，不应阻塞主路径。它的价值在长期复利，而不是单次任务的即时成功。

## 十二、Hook 与插件：扩展点优先于核心改动

### 解决的问题

Agent 项目一旦进入真实使用，就会出现大量定制需求：启动检查、工具审计、会话结束总结、外部记忆服务、不同压缩策略、平台级事件处理。如果每个需求都改核心代码，系统会快速失控。

### 设计方法

为生命周期事件和可替换能力定义扩展点。

- Hook 面向事件时刻：启动、会话开始、工具调用前后、会话结束。
- 插件面向可替换能力：记忆提供者、压缩策略、外部能力集成。
- 协调器负责调用插件，核心循环只依赖协调器接口。

### 模块协作

- Hook 注册表发现并触发事件处理器。
- 插件加载器扫描插件声明并收集提供者。
- 能力协调器负责 prefetch、sync、工具路由或策略调用。
- 插件异常被隔离，不能拖垮主流程。

### 可迁移经验

扩展点要围绕稳定生命周期和稳定接口设计，而不是围绕某个具体第三方服务设计。第三方服务会变，事件边界和能力契约应尽量稳定。

## 十三、自我演进：先度量，再优化，再部署

### 解决的问题

Agent 的技能文本、工具描述、系统提示词都会影响表现。人工修改这些文本常常靠感觉，难以证明变好了还是变差了。模型训练成本高，但很多行为改进其实可以通过优化上下文文本实现。

### 设计方法

把可编辑文本视为可优化对象。任何优化都走同一条路线：构建评估数据、测 baseline、收集反馈、生成改写候选、验证约束、在未见样例上复测，再部署。

### 模块协作

- 数据集构建器生成或收集任务样例。
- 评估器给当前版本打分，并产出可行动反馈。
- 约束门控限制体积、结构、语义漂移和回归风险。
- 优化器根据反馈生成候选版本。
- 部署层负责备份、审查、发布和回滚。

### 可迁移经验

自我演进不应直接等于“让模型改自己”。真正可控的演进需要评估集、硬约束、回滚路径和人工审查边界。没有度量的自动改写只是更快地产生不确定性。

## 方法论总结：几个高价值架构判断

### 1. 内核少变，边界多变

核心循环、状态分层、工具分发和安全恢复应保持稳定；平台协议、执行位置、外部服务和用户偏好会频繁变化，必须放在适配层、后端层或插件层。

### 2. 把“事实、方法、能力”分开

事实进入记忆，方法进入技能，能力进入工具。三者混在一起会导致 prompt 膨胀、技能失焦和工具滥用。

### 3. 把“模型想做”和“系统执行”分开

模型输出只是意图。工具分发、权限检查、执行环境、错误恢复和持久化共同决定这个意图是否以及如何落地。

### 4. 把“入口差异”和“任务逻辑”分开

CLI、消息平台、定时任务和外部触发都应转换到同一消息模型。业务任务不应知道用户来自哪里。

### 5. 把“运行路径”和“学习路径”分开

主任务路径追求完成和响应；学习路径负责记忆、技能、评估和优化。学习路径可以异步、可失败、可回滚，不能阻塞主任务。

### 6. 把“恢复策略”显式化

压缩、重试、续写、切换备用模型、拒绝执行都应是显式策略，而不是散落在异常处理里的临时补丁。

## 适合迁移到其他 Agent 项目的经验

- 先设计 harness，再选择模型。模型能力越强，越需要工程边界承接它的行动。
- 入口适配、工具能力、执行环境和持久化状态要正交拆分。
- 工具注册表是能力扩展的基础设施，不是语法糖。
- 长期记忆应少而精，完整会话应另行保存。
- 技能是可复用方法资产，应允许 Agent 在受控条件下读写。
- 高风险工具应有前置关卡，执行环境应隔离密钥和资源。
- 子 Agent 的价值在隔离和汇总，不在无限递归。
- 后台审视应筛选经验，而不是生成对话摘要。
- 插件和 hook 的异常必须隔离，不能破坏主流程。
- 自我优化必须先有评估和约束，再谈自动改写。

## 给后续 Agent 项目的设计指导原则

### 架构设计

1. 先画出五类边界：入口、核心循环、能力系统、执行环境、持久化状态。每个新需求必须落到其中一类，不能直接塞进核心循环。
2. 保持唯一核心循环。所有入口先统一为内部消息事件，再进入同一模型调用和工具执行路径。
3. 将系统状态按生命周期分层：即时上下文、完整会话、长期记忆、可复用技能、运行配置分别管理。
4. 任何可替换的外部依赖都通过接口接入：模型供应商、消息平台、执行后端、记忆服务、压缩策略都不应被硬编码进核心。

### 模块拆分

1. 工具模块只负责能力执行和自身描述；注册、筛选和分发交给工具注册表。
2. 技能模块只保存可迁移操作方法；不要把项目事实、用户画像或一次性结果写成技能。
3. 记忆模块只保存未来仍有用的精选信息；完整过程交给会话存储。
4. Prompt 组装模块只做来源合成、优先级、截断和缓存；不要承担业务逻辑。
5. 权限模块尽量靠近工具执行点，尤其靠近终端、文件写入、数据库和远程执行路径。
6. 适配器模块只处理协议差异，不处理 Agent 推理逻辑。
7. 执行后端模块只处理命令在哪里运行，不处理消息来自哪里。

### 协作关系设计

1. 入口适配器向内提交统一事件，向外发送平台回复；它不直接调用模型细节。
2. 核心循环调用工具注册表，不直接 import 具体工具实现。
3. 工具执行前后暴露 hook，使审计、日志和策略注入不需要改工具主体。
4. 子 Agent 只接收任务目标、必要上下文和受限工具集；返回摘要结果，不回灌完整轨迹。
5. 后台审视读取会话快照，写入记忆或技能，但不影响用户当前任务结果。
6. 插件由协调器调用；核心循环依赖协调器接口，不依赖具体插件。

### 方案取舍

1. 能用稳定接口解决的问题，不用条件分支堆在核心里解决。
2. 能异步沉淀的经验，不放在主任务路径里阻塞用户。
3. 能用文本技能表达的方法，不急着写成代码工具；只有稳定、频繁、高风险或需要系统权限的能力才沉淀为工具。
4. 能用会话记录审计的内容，不塞进长期记忆；记忆只保存未来决策会用到的内容。
5. 能通过适配器解决的平台差异，不让核心循环感知。
6. 能通过执行后端解决的环境差异，不让工具实现感知。
7. 能通过评估证明变好的优化，才允许部署；分数提升但约束失败的版本不应上线。
8. 对高风险能力，默认先限制再逐步放开；对低风险能力，默认先可用再逐步治理。

### 最小可行落地顺序

1. 做一个稳定核心循环：模型调用、工具调用、工具结果回写、迭代预算。
2. 加工具注册表：让新增工具不改循环。
3. 加会话持久化：保证重启、审计和历史检索。
4. 加分区 prompt：接入规则、记忆和技能索引。
5. 加上下文压缩和错误恢复：让长任务可持续。
6. 加权限关卡：先保护最高风险工具。
7. 加入口适配器和执行后端：让消息来源和执行位置正交扩展。
8. 加技能和后台审视：让经验能沉淀。
9. 加 hook、插件和评估优化：让系统可定制、可演进、可回滚。

这套顺序的核心理由是：先让 Agent 能稳定完成任务，再让它安全扩展，最后让它从使用中学习和优化。不要一开始追求多平台、自进化和插件生态；没有稳定内核，这些都会变成复杂度放大器。
