# s00: Architecture Overview (架构总览)

> 这一章是全仓库的地图。  
> 如果你只想先知道"整个系统到底由哪些模块组成、为什么是这个学习顺序"，先读这一章。

## Hermes Agent 和其他 agent 教学项目有什么不同

大多数 agent 教学项目教的是"一个人在终端里和 AI 对话"。

Hermes Agent 不是这样。它从一开始就被设计成一个**跨平台的自主 agent**：

- 同一套核心循环，可以通过终端对话，也可以同时挂在 Telegram、Discord、微信、飞书等十几个平台上
- 终端命令不一定在本地跑，可以在 Docker、SSH 远程机器或云端执行
- agent 可以自己创建和编辑技能文件，而不只是使用预先定义好的工具
- 所有对话都持久化到 SQLite，支持全文搜索，重启不丢
- 不绑定某一个模型提供商，通过 OpenAI 兼容接口支持 200+ 模型

这些不是后期加上去的功能。它们从一开始就影响了系统的核心设计。

所以学这套系统的时候，你面对的不是"一个简单循环加一堆工具"，而是一个有明确分层的平台架构。

## 这套仓库到底要还原什么

本仓库真正要还原的是：

- Hermes Agent 的核心分层是什么
- 为什么每一层存在
- 一条消息从用户发出到 agent 回复，中间经过了什么
- 关键状态存在哪里
- 什么设计决策让它能同时服务多个平台

我们追求的是：

**还原 Hermes Agent 独有的设计决策和架构分层，而不是通用的 agent 教学。**

## 三条阅读原则

### 1. 先只管 CLI，再管 Gateway

Hermes Agent 有两个入口：交互式终端（CLI）和多平台网关（Gateway）。

但你不需要一开始就同时学两个。阶段 1 到阶段 2 只讲 CLI 入口。到了阶段 3 才展开 Gateway。

因为 CLI 和 Gateway 最终调的是同一个核心循环。先把核心循环学透，后面接入多平台只是"换一个消息来源"。

### 2. 先只管本地执行，再管多种后端

同样的道理，终端命令可以在 Docker、SSH、云端执行，但阶段 1 只讲本地执行。

等你理解了工具系统的完整路径，再引入执行环境抽象只是"换一个命令跑的地方"。

### 3. 先做最小版本，每个机制都先停在教学边界

每一章都会先给一个最小但正确的实现，再告诉你"如果继续迭代应该补什么"。

不要在第一遍就追求把所有复杂度一起吃下去。

## Hermes Agent 的五层架构

这是 Hermes Agent 和大多数 agent 教学项目最不一样的地方：它不是"一个循环 + 一堆工具"，而是有明确的五层结构。

```text
┌─────────────────────────────────────┐
│ 入口层                               │
│ 用户从哪里来？                        │
│ CLI / Telegram / Discord / 微信 / …  │
└──────────────┬──────────────────────┘
               │
               v
┌─────────────────────────────────────┐
│ 核心循环层                            │
│ 模型思考 → 调工具 → 结果回写 → 继续   │
│ 同步循环，异步工具通过事件循环桥接      │
└──────────────┬──────────────────────┘
               │
               v
┌─────────────────────────────────────┐
│ 工具与智能层                          │
│ 工具自注册 / 记忆 / 技能 / 审批       │
│ 工具的添加不需要修改核心循环           │
└──────────────┬──────────────────────┘
               │
               v
┌─────────────────────────────────────┐
│ 执行环境层                            │
│ 命令在哪里跑？                        │
│ 本地 / Docker / SSH / 云端           │
└──────────────┬──────────────────────┘
               │
               v
┌─────────────────────────────────────┐
│ 持久化层                              │
│ SQLite 会话 / MEMORY.md / 技能文件    │
│ 对话和知识跨重启存活                   │
└─────────────────────────────────────┘
```

### 为什么是这五层

**入口层**：大多数 agent 只有一个终端入口。Hermes Agent 有两个入口——CLI 和 Gateway。Gateway 内部又有十几个平台适配器。入口层的作用是把不同来源的消息统一成同一种格式，然后交给核心循环。

**核心循环层**：这是系统心脏。无论消息从哪里来，最终都进入同一个循环：发给模型 → 处理工具调用 → 结果回写 → 继续。这个循环是同步的。异步工具（比如网络请求）通过一个持久化的事件循环桥接进来，而不是让整个循环变成 async。

**工具与智能层**：工具不是散落在各处的函数调用，而是一个自注册系统——每个工具文件在被导入时自动登记自己。记忆、技能、危险命令审批也在这一层。它们不修改核心循环的代码，而是在工具调用的前后插入逻辑。

**执行环境层**：终端命令不一定在本地跑。Hermes Agent 抽象出了一个执行环境接口，有本地、Docker、SSH、Modal、Daytona、Singularity 六种实现。上层工具不需要知道命令在哪里跑。

**持久化层**：对话历史存在 SQLite 里（不是文件），支持并发读写和全文搜索。记忆存在 MEMORY.md / USER.md 里。技能存在 skills 目录里。这些状态跨进程重启存活。

## 一条消息是怎么流动的

以 CLI 场景为例（最简单的路径）：

```text
1. 用户在终端输入一句话
2. CLI 创建一个 AIAgent 实例
3. AIAgent 组装 system prompt
   - 读 SOUL.md（人设）
   - 读 MEMORY.md / USER.md（记忆）
   - 读 HERMES.md 或 AGENTS.md（项目配置）
   - 拼上工具定义和技能清单
4. 把 messages + system prompt + tools 发给模型 API
   （通过 OpenAI SDK，所有提供商都走同一个接口）
5. 模型返回：
   - 如果是纯文本 → 显示给用户，结束
   - 如果是 tool_calls → 继续
6. 对于每个 tool_call：
   - 在工具注册表里查找 handler
   - 检查是否命中危险命令模式
   - 执行工具（在当前执行环境中）
   - 把结果以 tool 角色写回 messages
7. 回到步骤 4，继续下一轮
8. 循环结束后，把整个会话持久化到 SQLite
```

以 Gateway 场景为例（多了一层消息路由）：

```text
1. 用户在 Telegram 里发一条消息
2. Telegram 适配器收到消息，转成统一格式
3. Gateway 根据 chat_id 找到或创建一个会话
4. Gateway 创建一个 AIAgent 实例（传入历史消息）
5. 后面的步骤和 CLI 完全一样（步骤 3-8）
6. agent 的回复通过 Telegram 适配器发回给用户
```

关键洞察：**Gateway 场景和 CLI 场景的区别只在入口和出口，核心循环完全一样。**

## Hermes Agent 的四个关键设计决策

这四个决策是理解这套系统的钥匙。不理解它们，后面的章节会觉得"为什么要这样做"。

### 1. 为什么用 OpenAI SDK 作为唯一的 API 客户端

Hermes Agent 不绑定任何一个模型提供商。它用 OpenAI 的 Python SDK（`from openai import OpenAI`），通过设置不同的 `base_url` 接入不同的提供商。

这意味着从 OpenRouter 切换到 Anthropic 或本地端点，不需要改一行代码，只需要改配置。

这个决策影响了整个系统的消息格式：所有消息都用 OpenAI 格式（`role: user/assistant/tool`，`tool_calls`，`tool_call_id`）。

### 2. 为什么核心循环是同步的

大多数现代 Python 框架倾向于全异步。但 Hermes Agent 的核心循环是同步的 `def run_conversation()`，不是 `async def`。

原因很实际：

- 大部分工具（文件读写、终端命令）本身就是同步的
- 同步循环的错误处理和调试简单得多
- 少量异步工具（网络请求）通过一个持久化的事件循环桥接进来

这个"同步主体 + 异步桥接"的模式贯穿整个工具层。

### 3. 为什么工具用自注册而不是中心配置

每个工具文件在被 Python 导入时，自动调用注册表的 `register()` 方法登记自己。

导入链很清楚：注册表不依赖任何工具 → 工具依赖注册表 → 编排层导入所有工具触发注册 → 核心循环使用编排层。

这意味着添加一个新工具只需要写一个文件，不需要修改任何其他文件。

### 4. 为什么用 SQLite 而不是文件系统

因为 Gateway 场景下，多个平台的消息可能同时到达。SQLite 的 WAL 模式支持并发读写，FTS5 支持全文搜索历史会话。

文件系统方案在单用户 CLI 下够用，但在多平台并发场景下会出问题。

## 你真正需要掌握的关键状态

### messages — 工作记忆

当前对话的所有消息。不是聊天记录展示，是模型下一轮要读的输入。

### 工具注册表 — 能力目录

一张"工具名 → 处理函数"的映射表。工具自己注册，运行时查表分发。

### SQLite session — 持久记录

完整的会话元数据和消息历史。带全文搜索，支持并发。

### SOUL.md / MEMORY.md / HERMES.md — 长期上下文

三种不同的长期上下文来源：人设、记忆、项目配置。它们在每次对话开始时被组装进 system prompt。

### 执行环境 — 命令在哪跑

一个抽象接口，对上层透明。工具不关心命令是在本地还是在 Docker 里跑。

## 读者最容易卡住的地方

### "Gateway 和 CLI 到底是什么关系"

它们是同一个核心循环的两个不同入口。先只学 CLI，到阶段 3 再看 Gateway。

### "工具、技能、MCP 都是能力，有什么区别"

- 工具：硬编码在代码里的能力
- 技能：agent 运行时可以创建和编辑的能力文件
- MCP：通过外部协议接入的第三方能力

### "为什么有这么多配置来源"

因为 agent 可能在不同场景下运行（CLI 开发、Telegram 生产、Docker 隔离），每种场景需要不同的配置。配置系统用优先级链合并多个来源。
