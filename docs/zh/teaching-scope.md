# Teaching Scope (教学范围说明)

> 这份文档说明整个教学仓库到底教什么、不教什么。  
> Hermes Agent 是一个大系统。不是所有东西都适合从 0 到 1 教。

## 这份仓库的目标

**教开发者理解并重建一个跨平台自主 AI Agent 的核心架构。**

注意"跨平台"这个词。

大多数 agent 教学项目只教"一个人在终端里和 AI 对话"。Hermes Agent 从一开始就面对多平台、多执行环境、多模型提供商的现实。这些不是后期功能，它们影响了核心设计。

所以这套教学要覆盖的范围比单纯的"agent loop + tools"更大：

- 怎么让同一个循环服务 CLI 和 Gateway 两种入口
- 怎么让工具在不同执行环境里跑
- 怎么让 agent 自己管理技能文件
- 怎么用 SQLite 而不是文件系统做持久化
- 怎么通过 OpenAI 兼容接口支持任意模型

## 什么必须讲清楚

### 核心循环

同步循环怎么工作、为什么选择同步、异步工具怎么桥接进来、Gateway 模式下实例的生命周期是怎样的。

### 工具自注册

导入链是什么、为什么注册表不依赖工具、添加新工具为什么不需要改其他文件。

### 持久化

为什么用 SQLite 不用文件、WAL 模式解决什么问题、FTS5 解决什么问题、session 分裂和 parent_session_id 是怎么回事。

### 多来源 system prompt

SOUL.md、MEMORY.md、HERMES.md、AGENTS.md、.cursorrules 这些文件的优先级是什么、怎么组装成一条 system prompt、为什么要缓存 system prompt。

### Gateway 和适配器模式

Gateway 是什么、适配器基类定义了什么接口、消息怎么从平台特定格式变成统一格式、会话怎么按 chat_id 隔离。

### 执行环境抽象

BaseEnvironment 定义了什么接口、六种实现各自解决什么问题、工具层为什么不需要知道命令在哪跑。

### 技能生命周期

技能和工具的区别是什么、agent 怎么创建和编辑技能文件、技能怎么通过已有工具执行。

## 什么不要占主线篇幅

- **Nix flake / Docker 打包细节** — 怎么构建分发包不是核心架构
- **皮肤和主题引擎** — KawaiiSpinner 很可爱但不是教学重点
- **Nous Portal 订阅和计费** — 商业接线不属于核心机制
- **RL 训练管道** — batch_runner 和 trajectory_compressor 是独立子系统
- **各平台 API 的具体格式** — Telegram 的 inline keyboard、微信的 XML 消息结构这些是平台细节，不是架构知识
- **历史版本迁移逻辑** — config.py 里的迁移分支是技术债，不是设计决策
- **落地页和营销素材** — 不解释

## Hermes Agent 特有的教学难点

### 两个入口的心智负担

读者需要理解 CLI 和 Gateway 调的是同一个循环，但入口行为不同（CLI 是一个长实例，Gateway 是每消息一个新实例）。

教学策略：阶段 1-2 只讲 CLI。阶段 3 才展开 Gateway。先让核心循环稳固。

### 六种执行环境

工具发出的命令可能在本地、Docker、SSH、Modal、Daytona、Singularity 上跑。

教学策略：阶段 1-2 只讲本地。阶段 3 才引入环境抽象。先让工具系统稳固。

### 十七个平台适配器

Gateway 有十七个平台适配器，每个都有自己的消息格式和回调机制。

教学策略：只讲基类接口和统一消息格式。用一个具体平台做完整示例。其他平台说"同样的模式"。

### 多来源 system prompt

system prompt 由五六个来源组装，有优先级和缓存逻辑。

教学策略：先讲"有多个来源，拼在一起"。到 s04 再展开完整的组装逻辑和缓存策略。

## 每一章的推荐结构

1. `这一章要解决什么问题` — 先讲 WHY
2. `先解释几个名词` — 新概念先定义
3. `最小心智模型` — 一张图或一段伪代码
4. `关键数据结构` — 这个机制的状态长什么样
5. `最小实现` — 可以跑的代码
6. `Hermes Agent 的独特设计` — 这个机制在 Hermes 里有什么特别的
7. `初学者最容易犯的错` — 防坑
8. `教学边界` — 这一章先停在哪里

注意第 6 点。这是和通用 agent 教学不同的地方。每一章都应该点出 Hermes Agent 在这个机制上有什么独特的设计选择，而不只是讲通用概念。

## 维护者检查清单

- 这章有没有讲清楚"Hermes 在这个机制上和通用做法有什么不同"
- 有没有不小心写成了通用 agent 教学（和 learn-claude-code 换词没区别）
- 有没有在阶段 1 就提前引入 Gateway 或多执行环境的复杂度
- 代码示例用的是不是 OpenAI 格式（不是 Anthropic 格式）
- 有没有引用了不存在的类名或函数名（幻觉检查）

## 一句话

**这套教学的核心不是"怎么写一个 agent"，而是"怎么写一个能同时服务终端和十几个消息平台的 agent"。**
